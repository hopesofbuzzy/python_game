### 🎮 Концепт: `Vector Leap` (минималистичный 3D-платформер)
- **Визуал:** Low-poly, flat-shading/wireframe, разрешение `480×270` или `640×360`. Без текстур, без сложного света.
- **Геймплей:** Точное управление, прыжки, перемещение по платформам, простые триггеры/враги. Фокус на **физике, рендер-пайплайне и чистой архитектуре**.
- **Почему это работает для курса:** Псевдо-3D на CPU вынуждает явно реализовывать математический конвейер, а платформерная механика требует стабильной физики. Это естественным образом даёт алгоритмы разной сложности, которые логически связаны.

---
### 🔗 Алгоритмы, которые возникают естественно (не из таблицы, а из задачи)
| Алгоритм | Где и зачем | Сложность (по твоей рубрике) | Как связан с остальными |
|----------|-------------|------------------------------|--------------------------|
| **Перспективная проекция + Z-буфер / Painter's sorting** | Ядро рендера. Преобразует мировые координаты → экранные, решает перекрытие полигонов. | Сложный (×4) | Потребляет отфильтрованные полигоны из Culling. Результат → чистый `view`. |
| **Swept AABB (непрерывная коллизия)** | Игрок/объекты движутся с ускорением. Предотвращает провал сквозь тонкие платформы при падении/прыжке. | Сложный (×4) | Работает поверх сетки мира. Возвращает `collision_normal` и `time_of_impact` для физики. |
| **Fixed Timestep + полунеявный интегратор** | Гравитация и импульсы считаются строго 60 Гц. Прыжок одинаковый на 30 и 144 FPS. | Средний (×2) | Синхронизирует физику и рендер. Делает `Swept AABB` стабильным. |
| **Frustum Culling / пространственная сетка** | Отбрасывает полигоны за камерой или слишком далеко. Без этого CPU-рендер упрётся в 10 FPS. | Средний (×2) | Фильтрует данные перед проекцией. Использует ту же сетку, что и коллизии. |
| **FSM + Event Bus** | Состояния игрока (`idle → run → jump → fall → wallslide`). События: `landed`, `trigger`, `death`. | Лёгкий (×1) | Развязывает ввод, физику и UI. Чистая подписка/публикация. |

**Итог:** `2×4 + 2×2 + 1×1 = 13 баллов`.  
Это **максимум, который реально реализовать и защитить** за 1–2 месяца. Остальные баллы курс даст за архитектуру, чистоту кода и обоснование связей.

### 🏗 Архитектура под критерии (Model / View / SOLID)
```
project/
├── core/               # Движок, 0 зависимостей от игры
│   ├── loop.py         # FixedTimestep, main loop
│   ├── events.py       # EventBus (publish/subscribe)
│   └── math/           # Matrices, projection, swept_aabb, grid, utils
├── model/              # Чистая логика (0 pygame.draw/surface)
│   ├── entities.py     # Player, Platform, Enemy (данные + состояния)
│   ├── physics.py      # Integrator, collision resolver, ground check
│   └── world.py        # GameState, spatial grid, event routing
├── view/               # ТОЛЬКО проекция и отрисовка
│   ├── pipeline.py     # Culling → Project → Z-sort → Rasterize → blit
│   ├── camera.py       # Smooth follow, FOV, frustum bounds
│   └── ui.py           # Overlay, debug wireframe, FPS
└── main.py             # Инициализация, dependency injection, start
```

**Как это закрывает критерии экзамена:**
- ✅ `Model` не знает про пиксели, `Surface`, `blit`. Работает только с векторами, сетками, состояниями.
- ✅ `View` не хранит игровую логику. Получает `RenderBatch` (список треугольников с мировыми координатами и Z-глубиной) и рисует.
- ✅ `SOLID`: Интерфейсы `IPhysicsSolver`, `ISpatialIndex`, `IRenderPipeline`. Замена алгоритма не ломает систему.
- ✅ `DRY/KISS`: Одна функция проекции, один резолвер коллизий, один цикл обновления. Комментарии только `why`, константы в `config.py`.

### 📅 Реалистичный план на 6 недель
| Неделя | Фокус | Критерий готовности |
|--------|-------|---------------------|
| 1 | `Fixed Timestep` + базовая проекция + 1 куб + Z-буфер | Стабильный цикл, корректное перекрытие |
| 2 | `Swept AABB` + гравитация + прыжок + ground check | Игрок не проваливается, отскоки детерминированы |
| 3 | `Frustum Culling` + пространственная сетка | FPS стабильный при 50+ полигонах |
| 4 | `FSM` игрока + `EventBus` + простой триггер | Состояния переключаются плавно, UI реагирует |
| 5 | Интеграция, рефакторинг интерфейсов, профайлинг | Model/View чистые, зависимости явные |
| 6 | Тесты edge-cases, документация, схема потока данных | Готовая сдача: код + пояснение связей алгоритмов |

⚠️ **Главное правило:** Не добавляй контент. 1 игрок, 3–5 типов платформ, 1 триггер/враг, процедурная или статичная карта. Очки дают за **архитектуру, обоснование алгоритмов и чистоту кода**, а не за количество уровней.
